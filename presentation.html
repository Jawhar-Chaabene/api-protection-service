🛡️ API Protection Service

Real-time Malicious Traffic Detection & Blocking

A production-grade gRPC microservice implementing OWASP Top 10 security controls

Built with Go • MongoDB • Kafka • gRPC

🔴 The Problem

Modern APIs face sophisticated threats daily:

  • Injection attacks: SQL/NoSQL exploits targeting backend data
  • Unauthorized access: Users accessing resources they shouldn't
  • Rate-based attacks: DDoS and credential stuffing attempts
  • Privilege escalation: Attackers gaining admin-level access
  • Compliance violations: Unlogged security events for audit trails

Solution: A dedicated security service that validates, authorizes, and logs every request before it reaches your backend.

✅ The Solution

API Protection Service: A gRPC-based security decision engine

🔍 Input Validation

Strict schema validation on all incoming requests before processing

🚫 Injection Prevention

Multi-layer defense against NoSQL/SQL injection and command injection

👥 RBAC

Role-based access control with support for object-level authorization

⏱️ Rate Limiting

Per-client-IP token bucket limiting to prevent DoS attacks

Result: All requests verified before reaching your backend. DENY verdicts logged and alerted in real-time.

🔐 Layered Security Defense

Layer 1: Input Validation

Validates path format (must start with /), HTTP method whitelist, IP address format

Layer 2: Injection Prevention

Multi-defense: pattern detection, control character removal, length truncation

Layer 3: Authorization (RBAC)

Role-based access control: Anonymous → /api/public/*, User → /api/*, Admin → /admin/*

Layer 4: Rate Limiting

Per-client-IP token bucket (10 req/s, burst 20) prevents DoS attacks

Defense-in-depth: Even if one layer is bypassed, others provide protection

👥 RBAC Rules & Authorization

Role Allowed Paths Example
Anonymous /api/public/*, /health GET /api/public/docs → ALLOW
User /api/*, /users/{own_id} GET /api/users → ALLOW
GET /users/42 (owner) → ALLOW
Admin /admin/*, /users/* GET /admin/dashboard → ALLOW

Object-Level Authorization

Even with User role, users can only access their own resources:

  • ✅ GET /users/42 with x-user-id: 42 → ALLOW
  • ❌ GET /users/99 with x-user-id: 42 → DENY

📊 Request Processing Flow

1. Accept Request

gRPC server receives VerifyRequest with path, method, client_ip

2. Extract Metadata

MetadataInterceptor extracts x-roles, x-user-id, x-client-ip from headers

3. Rate Limit Check

RateLimitInterceptor checks token bucket per client IP

4. Validate Input

ValidateRequest checks schema: path format, method whitelist, IP validity

5. Check RBAC

DefaultRBAC verifies user roles grant access to resource

6. Log Decision

MongoStore saves verdict, reason, timestamp to audit trail

7. Alert (if DENY)

Kafka.PublishAlert sends real-time violation to security-alerts topic

8. Return Verdict

VerifyResponse with ALLOW/DENY verdict and reason

🧪 Testing & Validation

Test Methods

  • CLI Test Client: go run ./cmd/test-client - Automated 5-scenario verification
  • Postman gRPC: Interactive testing with gRPC reflection enabled

Core Test Cases

  • Public endpoint access (anonymous) → ALLOW
  • Health check endpoint → ALLOW
  • Users endpoint without role → DENY
  • Users endpoint with user role → ALLOW
  • Admin endpoint with admin role → ALLOW
  • Object-level access (own resource) → ALLOW
  • Object-level access (other user) → DENY

Validation Error Cases

  • Invalid path format (missing leading /) → DENY + validation error
  • Invalid HTTP method → DENY + validation error
  • Invalid IP address → DENY + validation error

🚀 Deployment & Configuration

Environment Variables

  • MONGODB_URI - MongoDB connection (default: mongodb://localhost:27017)
  • MONGODB_DB - Database name (default: api_protection)
  • KAFKA_BROKERS - Kafka brokers (default: localhost:9092)
  • RATE_LIMIT_RPS - Requests per second (default: 10)
  • RATE_LIMIT_BURST - Burst capacity (default: 20)

Service Startup

  • Ensure MongoDB is running and accessible
  • Ensure Kafka is running and accessible
  • Run: go run ./cmd/security-service
  • Expected output: "✅ Security Service listening on :50051"

Production Checklist

  • ✅ gRPC Reflection enabled for tooling (Postman, grpcurl)
  • ✅ MongoDB with authentication and TLS
  • ✅ Kafka with authentication and encryption
  • ✅ Port 50051 restricted to gateway only

📋 Audit & Compliance

MongoDB Audit Trail

Every verification decision is logged to api_protection.security_logs:

  • Request details: path, method, client IP
  • Decision: ALLOW or DENY verdict
  • Reason: Detailed explanation if denied
  • Timestamp: UTC timestamp of decision

Kafka Real-Time Alerts

All DENY verdicts published to security-alerts topic:

  • Enables real-time SIEM integration
  • Supports incident response automation
  • Allows security analytics and dashboards

proto/ - gRPC definitions

  • security.proto - Service and message contracts
  • genProto/ - Generated Go code
