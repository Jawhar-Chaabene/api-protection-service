Executive Summary

API Protection Service is a production-grade Go microservice that provides real-time detection and blocking of malicious traffic compliant with OWASP Top 10 vulnerabilities.

Core Architecture: The service implements a layered security architecture using gRPC as the communication protocol between an HTTP Gateway and a dedicated Security Service. All security decisions are logged to MongoDB for audit trails, and critical violations are published to Kafka for real-time alert processing.

Key Features: Input validation, role-based access control (RBAC), object-level authorization, per-client rate limiting, injection attack detection, cryptographic hashing, and asynchronous alert publishing.

4.2 Service Layer - Core Business Logic

📄 internal/service/security.go
Purpose:

Core security verification and authorization decision engine

Contains:
  • SecurityService struct with dependencies:
    • store - Persistence layer (MongoDB)
    • alertPublisher - Event publisher (Kafka)
    • rbac - Authorization checker
  • Verify() method - The core verification logic:
    • Step 1: Input Validation
      • Validates path format, HTTP method, IP address
      • Fail-fast: returns DENY immediately if validation fails
      • Provides detailed error message explaining validation failure
    • Step 2: RBAC + Object-Level Authorization
      • Extracts user ID and roles from context
      • Checks if user's roles grant access to requested resource
      • Supports object-level authorization (users can only access own profile)
      • Returns DENY with reason if authorization fails
    • Default Action: ALLOW if both validation and authorization pass
  • finishVerify() helper - Audit and alerting:
    • Saves all decisions to MongoDB audit log
    • For DENY verdicts: publishes alert to Kafka asynchronously
    • Handles errors gracefully without affecting verdict
Role:

The critical business logic engine. Makes the fundamental allow/deny decision by orchestrating validation and authorization checks. Ensures all decisions are logged for compliance and triggers real-time alerts for security violations.

📄 internal/service/rbac.go
Purpose:

Role-Based Access Control (RBAC) and object-level authorization policies

Contains:
  • RBACChecker interface - Defines authorization contract:
    • CanAccess(userID, roles, resourcePath, method) → (allowed, reason)
  • DefaultRBAC - Implementation with three role-based rules:
    • Anonymous role: Access to /api/public/* and /health
    • Admin role: Access to /admin/* and /users/*
    • User role:

    • extractResourceID() - Parses ID from path for object-level checks
Role:

Implements authorization policies. Determines which users with which roles can access which resources. Supports both role-based access (coarse-grained) and object-level access control (fine-grained). Can be extended with different RBAC implementations.

5. pkg/ Directory - Reusable Utilities

Contains general-purpose packages that could potentially be reused in other projects.

📄 pkg/crypto/argon2.go
Purpose:

Secure password/secret hashing using Argon2id algorithm

Contains:
  • Argon2id parameters (hardened):
    • Time: 1 iteration (minimal, suitable for login paths)
    • Memory: 64MB - High memory cost resists GPU/ASIC attacks
    • Parallelism: 4 threads
    • Key length: 32 bytes (256 bits)
    • Salt length: 16 bytes (128 bits random)
  • HashSecret(secret []byte) - Hashing function:
    • Generates cryptographically random salt
    • Computes Argon2id hash with salt
    • Encodes result in standard format: $argon2id$v=19$m=65536,t=1,p=4$<salt_b64>$<hash_b64>
  • VerifySecret(secret, encodedHash) - Verification function:
    • Decodes stored hash to extract salt and parameters
    • Recomputes hash with provided secret and stored salt
    • Uses constant-time comparison to prevent timing attacks
  • decodeHash() - Hash format parser
Role:

Provides secure credential storage. Argon2id is the current OWASP-recommended password hashing algorithm, resistant to GPU and ASIC attacks. Used for storing API keys, service credentials, and user passwords. The constant-time comparison prevents attackers from using timing analysis to guess correct hashes.

📄 pkg/kafka/producer.go
Purpose:

Kafka producer for publishing real-time security alert events

Contains:
  • AlertRecord struct - Alert message payload:
    • path, method, client_ip - Request details
    • verdict, reason - Decision and explanation
    • timestamp - When alert was generated (UTC)
    • Serializes to JSON automatically
  • Producer struct - Thread-safe Kafka wrapper:
    • Wraps franz-go client
    • NewProducer(brokers []string) - Constructor
    • PublishAlert(ctx, record) - Synchronous publish with error handling
    • Close() - Graceful shutdown
  • Topic: security-alerts - Kafka topic for all alert events
Role:

Decouples security verification from alert processing. Asynchronously publishes denial events to Kafka where other systems can subscribe and react (SIEM ingestion, incident response automation, real-time dashboards). Enables scalable event-driven security operations.

📄 pkg/sanitize/sanitize.go
Purpose:

Input sanitization to prevent injection attacks

Contains:
  • injectionPattern regex - Detects common injection attack prefixes:
    • $, {}, \, " - MongoDB/SQL operator characters
  • Metadata(value string) - Sanitization function applying three defenses:
    • Pattern rejection: Rejects values matching injection pattern → returns empty string
    • Character filtering: Removes control characters (0x00-0x1F, 0x7F)
    • Length truncation: Limits to 256 characters maximum
  • Defense-in-depth approach: Multiple layers ensure that even if one is bypassed, others catch attacks
Role:

Defense-in-depth sanitization. Provides multiple layers of protection (pattern detection, character filtering, truncation) against injection attacks in metadata. Applied to all gRPC metadata before use in authorization decisions. Particularly effective against NoSQL/MongoDB injection.

7.2 Request Processing Flow

Stage Component Action Decision
1. Receive gRPC Server Accept VerifyRequest Continue
2. Metadata Extract MetadataInterceptor Extract & sanitize headers Continue
3. Rate Limit RateLimitInterceptor Check token bucket per IP ALLOW or ResourceExhausted
4. Validate Input ValidateRequest() Check path, method, IP format DENY (validation error) or Continue
5. Check RBAC DefaultRBAC Verify user roles grant access DENY (no access) or ALLOW
6. Log Decision MongoStore.SaveLog() Insert audit record Continue
7. Alert (if DENY) Kafka.PublishAlert() Publish to security-alerts topic Async
8. Return gRPC Server Send VerifyResponse ALLOW or DENY
